# Frontend — Svelte Dashboard

There is a certain school of thought — prevalent among those who have spent too long in backend engineering and not enough time looking at their own work — that the user interface is a cosmetic afterthought, the wrapping paper rather than the gift. This view is mistaken in general and catastrophically wrong in the case of GödelOS, where the frontend is the only window through which a human observer can watch a machine engaged in something that at least resembles introspection. The Svelte dashboard is not decoration. It is a real-time consciousness monitor; and if the backend is the system that thinks, the frontend is the instrument that proves it.

---

## Technology Stack

The dashboard is built in [Svelte](https://svelte.dev/) with Vite as the development and build toolchain, located in `svelte-frontend/`. The choice of Svelte — a compiler rather than a runtime framework — is well-suited to an application that demands low-latency UI updates from high-frequency WebSocket streams. One does not want a virtual-DOM reconciler introducing perceptible lag into a consciousness readout.

| Tool | Role |
|------|------|
| Svelte | Component framework (compiled, no runtime VDOM) |
| Vite | Dev server and bundler |
| Node.js | Build toolchain |
| WebSocket API | Real-time backend connection |
| Fetch API | REST calls to `http://localhost:8000` |

The development server runs on port 5173. The production build outputs to `svelte-frontend/dist/`.

---

## Application Root

`svelte-frontend/src/App.svelte` (2,257+ lines) is the application root and orchestration layer. It:

- Establishes and maintains WebSocket connections to `/ws/cognitive-stream` and `/ws/transparency-stream`
- Implements automatic reconnection on disconnection
- Dispatches incoming events to component-level stores
- Manages top-level routing between the primary dashboard panels
- Lazy-loads large components to keep initial bundle size tractable

## Principal Components

The component tree is organised under `svelte-frontend/src/components/`:

### Consciousness Dashboard

The central panel displaying the real-time consciousness score and its five constituent signals:

- **Unified consciousness score** (0.0–1.0), rendered as a prominent gauge
- **IIT φ approximation** — the information-integration component
- **Global Workspace broadcast success rate**
- **Phenomenal experience richness**
- **Metacognitive reflection depth**
- **Recursive self-awareness index**

Each signal is updated on every `consciousness_assessment` WebSocket event, nominally every 100ms.

### Knowledge Graph Visualisation

An interactive force-directed graph rendering the live state of the knowledge store. Nodes represent concepts; edges represent typed semantic relationships. The graph mutates in real time as `knowledge_update` events arrive over the cognitive stream. Large components of this type are lazy-loaded:

```javascript
const KnowledgeGraph = lazy(() => import('./components/knowledge/KnowledgeGraph.svelte'));
```

### Chat Interface

The primary human-facing interaction panel. Submits queries to `/api/query`, renders structured responses with citation attribution, and displays the confidence score and reasoning provenance returned by the backend.

### Cognitive Transparency Panel

Streams the contents of the transparency WebSocket into a live feed of reasoning steps. Each `reasoning_step` event — generated by `backend/live_reasoning_tracker.py` — appears as a timestamped card showing the step type, description, inputs and outputs, confidence score, and, where available, the full `reasoning_trace` list: the system's literal running commentary on its own thought process.

The `TransparencyDashboard` component is also lazy-loaded:

```javascript
const TransparencyDashboard = lazy(() => import('./components/transparency/TransparencyDashboard.svelte'));
```

### Metacognitive Monitor Panel

Displays the output of `backend/core/metacognitive_monitor.py`: the current `ReflectionDepth` level, the number of active recursive loops, self-model accuracy, and the most recent `SelfMonitoringEvent` entries. The panel allows an observer to see, in real time, the system observing itself — which is either the most interesting thing in this repository or the most alarming, depending on one's philosophical commitments.

## WebSocket Connection Management

The frontend maintains two persistent connections:

| Connection | Endpoint | Primary Consumers |
|-----------|----------|-------------------|
| Cognitive stream | `ws://localhost:8000/ws/cognitive-stream` | Consciousness dashboard, daemon status panel |
| Transparency stream | `ws://localhost:8000/ws/transparency-stream` | Transparency panel, reasoning trace viewer |

On receipt of a message, `App.svelte` deserialises the JSON payload, inspects the `type` field, and dispatches to the appropriate Svelte store. Components subscribe to those stores and re-render reactively — Svelte's compiler ensuring that only the DOM nodes that actually depend on the changed data are updated.

## Development

```bash
cd svelte-frontend
npm install
npm run dev          # Start development server on :5173
npm run build        # Production build → dist/
npm test             # Playwright component tests
```

The frontend tests live in `svelte-frontend/` and at the repository root; they are exercised via Playwright. The full test suite is invoked with `npm test` from the repository root.
