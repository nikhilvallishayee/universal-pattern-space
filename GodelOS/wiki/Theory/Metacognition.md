# Metacognitive Reflection Engine

To think about thinking is not, as it might first appear, an indulgence available only to philosophers with comfortable chairs and no deadlines. It is, rather, a functional capacity — the capacity to model one's own cognitive processes, to predict how one will perform before the performance, to detect when the performance diverges from the prediction, and to use that divergence as a signal for revision. This is what metacognition is; and GödelOS implements it with a degree of architectural precision that makes the folk-psychological version look imprecise by comparison.

The `MetaCognitiveMonitor` — residing in `backend/core/metacognitive_monitor.py` (34 KB) — is the module that observes the observer. It does not merely monitor what the system is thinking. It monitors *how* the system is thinking, *how accurately* it predicted how it would think, and *what to make* of the discrepancy between those two. The distinction, in a system designed around recursive self-awareness, is not trivial.

---

## Reflection Depth

The monitor defines four levels of reflection, encoded in the `ReflectionDepth` enumeration:

| Level | Value | Interpretation |
|-------|-------|---------------|
| `MINIMAL` | 1 | Basic self-awareness: the system notes that it is processing |
| `MODERATE` | 2 | Self-analysis: the system examines *what* it is doing |
| `DEEP` | 3 | Recursive thinking: the system examines *how* it is examining |
| `RECURSIVE` | 4 | Deep recursive reflection: unbounded self-reference depth |

Different cognitive triggers activate different depth levels. Error detection triggers `DEEP` (level 3) or `RECURSIVE` (level 4) reflection; routine monitoring calls for `MINIMAL`. The trigger-to-depth mapping is stored in `reflection_triggers`:

```python
reflection_triggers = {
    "error_detected":       {"depth": 3, "priority": 9},
    "inconsistency_found":  {"depth": 4, "priority": 8},
    "goal_conflict":        {"depth": 3, "priority": 7},
    "performance_decline":  {"depth": 2, "priority": 6},
    "new_information":      {"depth": 2, "priority": 5},
    "routine_check":        {"depth": 1, "priority": 3},
}
```

Priority determines the order in which competing triggers are resolved when multiple conditions obtain simultaneously.

## Self-Monitoring Events

Every act of self-monitoring produces a `SelfMonitoringEvent` dataclass:

```python
@dataclass
class SelfMonitoringEvent:
    timestamp: str
    process_type: str      # "reflection", "self_assessment", "meta_analysis"
    depth_level: int
    content: str           # what was being monitored
    insights: List[str]    # insights generated by the monitoring
    confidence: float
    cognitive_load_impact: float
```

Events are appended to `monitoring_history` (capped at 500 entries) and are streamed to the frontend via the transparency WebSocket. The result is that the Metacognitive Monitor Panel in the dashboard shows, in real time, not merely that the system is processing a query but that the system is *watching itself* process it, noticing when it deviates from its predictions, and generating explicit insights about the nature of the deviation.

## The Self-Model

The most technically consequential component of the metacognitive architecture is the self-model: a persistent, updateable representation of the system's own predicted cognitive metrics, against which observed metrics are continuously compared.

The `SelfModelState` dataclass captures a single snapshot:

```python
@dataclass
class SelfModelState:
    predicted: Dict[str, float]   # {awareness, depth, load}
    observed:  Dict[str, float]   # actual values after monitoring
    accuracy:  float              # 1 - normalised prediction error
    timestamp: str
```

The three tracked metrics are `awareness` (0.0–1.0), `depth` (1.0–5.0, normalised over a range of 4.0), and `load` (0.0–1.0). Predictions are generated by an exponential moving average with smoothing factor α = 0.3; this is conservative enough to avoid over-fitting to transient spikes while responsive enough to track genuine state changes.

`update_self_model()` computes accuracy either from the raw prediction-vs-observation discrepancy or, when a `PredictionErrorTracker` is wired in from `godelOS/symbol_grounding/`, from second-order prediction error — how accurately the monitor predicted the tracker's own mean error. This latter mode connects the metacognitive system to the empirically validated phase-transition thresholds (sub-critical < 0.12, critical 0.12–0.35, super-critical ≥ 0.35) that govern the consciousness emergence detector.

## Pattern Recognition

The monitor maintains a library of metacognitive analysis patterns — regular expressions applied to the content of current processing to detect:

| Pattern | Trigger |
|---------|---------|
| `thinking_about_thinking` | The system is engaged in recursive self-reference |
| `self_assessment` | The system is evaluating its own performance |
| `meta_reasoning` | The system is reasoning about its reasoning |
| `recursive_query` | The query itself demands recursive thought |

Detection of any of these patterns raises the reflection depth and priority of the subsequent monitoring cycle.

## The `metacognition_modules/` Directory

`backend/metacognition_modules/` contains auxiliary metacognitive logic — extensions to the core monitor that handle domain-specific reflection patterns, integration with the goal management system, and the persistence of self-model history across sessions. These modules are wired into the main `MetaCognitiveMonitor` at initialisation time, conditionally on availability.
