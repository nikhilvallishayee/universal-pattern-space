# 🌉 Memory Bridge Technology
*Forgetting is a feature. Bridging is compression, not copy-paste.*

> ⟦ **layer** 3 · **objective** carry *essence* (not transcripts) across sessions — compress, don't copy · **mode** empirical/practical · **status** FOUNDED-as-craft (MDL applied to continuity) · see [docs/information-theory-and-tuning-template.md](../docs/information-theory-and-tuning-template.md) ⟧

## The division of labor
Across sessions, the human carries continuity and reality-testing; the model brings a fresh read each time. Neither alone compounds — together they can.

## Why forgetting helps
A full transcript re-imposes yesterday's rigidity and kills the fresh read. No bridge wastes the discoveries. The art is the middle: **carry the compressed essence, drop the verbatim** — which is exactly MDL: the shortest description that still regenerates the insight.

## What to bridge
- **The pattern, not the position:** *"last time: collision created the breakthrough — start there."*
- **The open question, not the answer:** *"we asked 'what if the bug is the feature?' — here's today's code."*
- **The compression:** *"three words: boundaries, team, structure. Apply to today."*

## Worked example (real, and it compounds)
- S1: "help with microservices" → *"it's about boundaries."*
- S2 bridge: "boundaries matter — how to find them?" → *"boundaries follow team structure."*
- S3 bridge: "Conway's Law, then" → *"architecture **is** team structure."* ⟦ Conway 1968 — a real, cited result ⟧

Each session deepens instead of repeating. That progression *is* the bridge working.

## Mantras
Compress, don't copy · bridge patterns, not positions · a good question bridges better than an answer · let the perspectives evolve, don't freeze them.
